# Requirements

## 1. 文書の目的
本書は、`feature-separate-medication-schedule` の開発における要求定義を明確化する。
実装方式は定義せず、なぜ作るか・誰の課題を解決するか・何を満たすべきかを整理する。

## 2. 背景
- 現行アプリは `Medication` が服用時間帯・時刻を保持しており、同一タイミングで飲む複数薬の管理が薬件数分に分散している。
- 実利用では「朝のセット」「夜のセット」のように、時間単位で複数薬を扱う運用が中心である。
- そのため、薬マスタと運用スケジュールを分離しない限り、登録負荷・変更漏れ・通知整合の問題が解消しにくい。

## 3. 対象ユーザーと課題
### 3.1 主対象ユーザー
- 毎日複数薬を服用する個人ユーザー
- 家族の服薬を補助管理するユーザー

### 3.2 解決すべき課題
- 同じ時刻に飲む薬を複数登録した際、時刻変更を薬ごとに修正する必要がある。
- 通知・一覧・ウィジェットが薬単位で、1回の服薬行動を直感的に扱いづらい。
- 履歴が薬単位のみで、スケジュール単位の実行確認が難しい。

## 4. 業務要求
- BRQ-1: 服薬実態に合わせて「時間単位（スケジュール単位）」で管理できること。
- BRQ-2: 薬マスタは再利用可能な基礎情報として独立管理できること。
- BRQ-3: 1回の操作で、スケジュールに紐づく複数薬の服用記録を残せること。
- BRQ-4: 既存ユーザーの薬データ/履歴データを移行し、利用継続を阻害しないこと。

## 5. 機能要求
- FR-1: 一覧画面はスケジュール中心（名称・時刻/時間帯・対象薬）で表示する。
- FR-2: スケジュール登録/編集画面を提供し、薬を1件以上選択して紐づけられる。
- FR-3: 薬マスタ登録/編集画面をスケジュール登録画面と分離する。
- FR-4: スケジュール実行（飲んだ）で紐づく薬を同一日時で一括記録する。
- FR-5: 手動追加時はスケジュール単位での追加を可能にし、未来日時入力を禁止する。
- FR-6: 通知はスケジュール単位で発火し、該当スケジュール情報を提示する。
- FR-7: 履歴画面はスケジュール起点で参照でき、必要時に薬内訳を表示できる。
- FR-8: 薬一覧ウィジェットはスケジュール中心に変更し、ウィジェットから実行可能とする。
- FR-9: 既存の重複防止ルール（同一対象・同一時刻）と5分再実行抑止を維持または拡張する。

## 6. 非機能要求
- NFR-1: Android 7.0+（minSdk 24）で動作すること。
- NFR-2: オフライン完結（外部API/サーバ連携なし）を維持すること。
- NFR-3: 通知は可能な限り正確な時刻配信を維持すること（AlarmManagerの正確アラーム優先）。
- NFR-4: 端末再起動後に通知スケジュールを復元すること。
- NFR-5: DB更新後のウィジェット反映遅延を最小化すること。
- NFR-6: 既存データ移行時に致命的なデータ欠損を起こさないこと。

## 7. 制約
- C-1: 実装対象はローカルAndroidアプリ内に限定し、認証・マルチユーザーは対象外。
- C-2: 既存の通知権限フロー（Android 13+）を維持する。
- C-3: 既存履歴の参照可能性を維持する（移行後も履歴表示可能）。
- C-4: 仕様駆動文書は `docs` / `specs` 配下で管理し、アプリソース配下には作成しない。

## 8. 優先順位
### P0（必須）
- スケジュールと薬マスタの分離
- スケジュールへの複数薬紐づけ
- スケジュール一括記録
- 通知/履歴/ウィジェットのスケジュール中心化

### P1（高）
- 既存データ移行の自動化
- 手動追加・重複防止ルールのスケジュール対応

### P2（中）
- UI文言・導線の最適化
- 将来拡張（曜日別、休薬設定）を見据えたデータ構造準備

## 9. 非対象
- クラウド同期
- ユーザーアカウント/認証
- 医療機関システムとの連携
- 服薬以外の健康ログ管理
